# Ejercicio 1.5 — Diseño de tabla de eventos de navegación web

## Particionamiento

Partición **diaria por `event_date`** (DATE derivada del timestamp del evento). Con 20-50M eventos/día, la partición diaria permite que cualquier query sobre un rango de fechas escanee solo las particiones necesarias. El 80% de las queries consultan los últimos 7 días, lo que refuerza esta decisión: el motor lee máximo 7 particiones en vez de la tabla completa.

**¿Por qué diaria y no mensual?**
- Partición mensual (30-31 particiones) sería demasiado gruesa para este volumen. Una query sobre 7 días leería igual que una sobre 2 días (la partición de febrero completa).
- Partición diaria (365 particiones/año) es granular y eficiente: acceso selectivo sin fragmentación excesiva.
- BigQuery maneja 365 particiones sin penalización; es el estándar de la industria.

## Clustering

Columnas en este orden: **`event_type` → `user_id` (o `session_id`)**.

**Primero `event_type`** porque es la dimensión más frecuente en los filtros de análisis:
- Filtros por tipo de evento: "páginas vistas" (page_view), "clicks" (click), "formularios completados" (form_submit).
- El clustering ordena la tabla físicamente por este valor, agrupando el mismo tipo juntos.

**Segundo `user_id`** para optimizar queries de análisis de comportamiento por usuario:
- Queries: "¿Cuántas acciones hizo el user 12345?" o "comportamiento del usuario dentro de clicks".
- El clustering secundario agrupa usuarios dentro de cada tipo, multiplicando eficiencia.

**El orden importa:** El clustering es eficiente de izquierda a derecha, como un índice compuesto en bases de datos tradicionales. Invertir el orden reduciría beneficio.

## Control de costes

Tres medidas:

1. **Siempre filtrar por la columna de partición** en el WHERE para evitar full table scans.
   -  `WHERE event_date BETWEEN '2025-01-01' AND '2025-01-07'` → lee 7 particiones
   -  `WHERE CAST(event_timestamp AS DATE) BETWEEN ...` → full table scan, sin beneficio de particionamiento

2. **Seleccionar columnas específicas** en vez de `SELECT *`
   - En tablas columnares (como BigQuery) el coste es proporcional a las columnas leídas.
   -  `SELECT user_id, event_type, event_timestamp` → solo 3 columnas
   -  `SELECT *` → todas las columnas, incluidas las no necesarias (geolocation, device_specs, etc.)

3. **Crear tabla materializada** (o vista agregada) de los últimos 7 días para queries frecuentes.
   - Mantener tabla de snapshot diario con métricas pre-agregadas (conteos por tipo, usuario, etc.)
   - Reducir volumen procesado del 100% al ~2% de la historia total.
   - Refrescar cada noche con datos nuevos (proceso ETL).

## Streaming vía Pub/Sub

**Tabla de tipo STREAMING** en BigQuery (antiguo insert streaming) o preferiblemente **BigQuery Storage Write API** en modo committed:
- Ofrece mayor throughput (100k+ inserciones/segundo).
- Garantías exactly-once: sin duplicados, sin pérdidas.
- Los datos llegan al buffer de streaming y se consolidan en el almacenamiento particionado.

**Arquitectura moderna complementaria:**
- **Bigtable** para acceso de baja latencia en tiempo real (< 10ms).
- **BigQuery** para análisis histórico y reportes (particionado, clusterizado).
- **Pub/Sub** como capa de desacoplamiento entre producer (aplicación web) y consumers (Bigtable, BigQuery).

```
Aplicación Web → Pub/Sub → {BigQuery Storage Write API, Bigtable}
```

## Impacto del patrón de consulta (80% últimos 7 días)

Este patrón refuerza todas las decisiones anteriores:

1. **Partición diaria:** 80% de queries leen máximo 7 particiones. Muy eficiente.
2. **Tabla materializada caliente:** Mantener snapshot de últimos 7 días en almacenamiento Standard (hot tier) para queries rápidas.
3. **Archivo histórico frío:** El 20% de queries de histórico (> 7 días) pueden usar almacenamiento Nearline/Coldline para reducir costes (10-20x más barato).
4. **Política de lifecycle automática:**
   ```
   - 0-7 días   → Standard (hot)
   - 8-30 días  → Nearline (warm)
   - 31+ días   → Coldline (cold)
   ```
   BigQuery aplica esta política automáticamente tras configurarla.

**Impacto económico:** Reducción de 60-70% en costes de almacenamiento sin sacrificar rendimiento en las queries más frecuentes.

## Resumen de decisiones de diseño

| Componente | Decisión | Justificación |
|------------|----------|---------------|
| Particionamiento | Diario (`event_date`) | Volumen 20-50M/día; 7 particiones para 80% de queries |
| Clustering | `event_type`, `user_id` | Filtros frecuentes; orden importa (izq→dcha) |
| Filtering | Obligatorio en partición | Evita full table scans; ahorra costes |
| Proyección | Columnas específicas | Coste proporcional a ancho de columnas |
| Agregación | Tabla materializada 7 días | Snapshot diario pre-procesado |
| Streaming | Write API (committed) | Throughput alto + exactly-once |
| Lifecycle | 0-7 Standard, 8-30 Nearline, 31+ Coldline | Reduce costes 60-70% para histórico |
